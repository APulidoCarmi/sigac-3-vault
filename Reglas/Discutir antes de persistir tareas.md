# Regla — Discutir antes de persistir tareas

No escribir a `Tareas.md`/`Planes/` apenas se formula una conclusión
durante una sesión de análisis. Proponer primero en el chat (redacción
completa, lista para revisión), esperar confirmación explícita del
usuario o que pida directamente "genera las tareas", y solo entonces
escribir al baúl.

**Por qué:** surgió en la sesión [[2026-07-21 - Análisis de flujo de referencias (visita Querétaro)]]
cuando el usuario rechazó una edición a mitad de discusión ("vamos a
platicarlo primero"). Se confirmó de nuevo en la sesión del 2026-07-22
(mismo hilo Querétaro): cada una de las ~13 tareas nuevas se redactó,
corrigió y re-redactó varias veces en el chat —a veces varias rondas por
una sola tarea— antes de escribirse a `Tareas.md`. Sin esta disciplina,
una sesión larga con mucha iteración habría dejado el archivo con
versiones a medio corregir.

**Cómo aplicarla:**
- Al llegar a una conclusión de diseño o alcance, redactarla completa en
  el chat como si ya fuera la línea final de `Tareas.md`/`Planes/`.
- Si el usuario pide un ajuste, no reescribir el archivo todavía — ajustar
  la propuesta en el chat y volver a mostrarla.
- Escribir al archivo solo cuando el usuario dé luz verde explícita
  ("así queda", "genera las tareas", "agrégala").
- Aplica también a decisiones de estructura del propio archivo (ej.
  reestructurar `Tareas.md` en secciones): confirmar el enfoque antes de
  reescribir el archivo completo.
